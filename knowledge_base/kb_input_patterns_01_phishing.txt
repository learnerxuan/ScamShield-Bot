PURPOSE
================================================================================
This file supports the "paste suspicious message or link" feature for phishing-
style scam checks. It is designed for rule-based extraction from pasted SMS,
email text, chat text, and visible link wording.

This file is for:
- message-level red flag extraction
- follow-up question design
- rule-based risk escalation

This file is NOT for:
- live website inspection
- definitive URL reputation checking
- malware analysis
- proving a domain is safe

SECTION 2: COMMON PHISHING WORDING PATTERNS
================================================================================

ACCOUNT VERIFICATION / ACCOUNT PROBLEM
- "verify your account"
- "unlock your account"
- "secure your account"
- "account suspended"
- "account blocked"
- "update your details"
- "confirm your identity"

URGENCY / THREAT
- "act now"
- "within 24 hours"
- "immediately"
- "final warning"
- "legal action"
- "your account will be closed"
- "transaction will be cancelled"

REWARD / REFUND / PRIZE
- "claim your refund"
- "you won a prize"
- "tax refund"
- "reward pending"
- "bonus claim"

CREDENTIAL / OTP HARVESTING
- "enter OTP"
- "confirm OTP"
- "verify password"
- "update PIN"
- "banking password"
- "login again to continue"

================================================================================
SECTION 3: LINK AND URL PATTERNS EXTRACTABLE FROM TEXT
================================================================================

- message contains a click prompt:
  - "click here"
  - "tap to verify"
  - "open this link"
  - "log in here"
- visible domain mismatch or brand misspelling
- shortened-link services such as bit.ly or tinyurl
- visible HTTP instead of HTTPS
- legitimacy language such as:
  - "official verification portal"
  - "secure portal"
  - "official support page"

================================================================================
SECTION 4: IMPERSONATION PATTERNS
================================================================================

Common impersonated entities:
- banks
- government agencies
- tax/refund authorities
- telcos
- courier services
- popular online services

Message-level impersonation cues:
- sender claims to be a bank or regulator
- sender name does not match expected domain
- message gives a hotline or support number not known to the user
- generic greeting instead of the user's name

================================================================================
SECTION 5: EXTRACTABLE INDICATORS FOR THE BOT
================================================================================

- hasLink
- hasUrgency
- requestsOtp
- requestsPasswordOrPin
- requestsSensitiveInfo
- claimsAccountProblem
- claimsRewardOrRefund
- senderMismatch
- suggestsUnofficialHotline
- requestsAppInstall

================================================================================
SECTION 6: POST-INCIDENT INDICATORS
================================================================================

Escalate immediately if user says they:
- already clicked the link
- already entered login credentials
- already entered OTP
- already called the suspicious number
- already downloaded or installed an app
- already noticed unauthorized account activity

================================================================================
SECTION 7: RULE-BASED USE IN THE BOT
================================================================================

Recommended flow:
1. User pastes message or link.
2. Bot extracts phishing-style indicators from the pasted content.
3. Bot asks follow-up questions:
   - Did it ask for OTP or password?
   - Did you already click?
   - Did you enter anything?
   - Did the sender or site look suspicious?
4. Bot calculates final risk from both the pasted content and user answers.

================================================================================
SECTION 8: LIMITATIONS
================================================================================

Do not claim these from pasted text or visible link alone:
- that a URL is definitely malicious
- that a URL is definitely safe
- that a domain really belongs to the claimed institution
- that a shortened link resolves to a malicious site without expansion
- that a website contains malware without technical inspection

Therefore:
- The feature must be framed as a preliminary rule-based assessment.
- The bot should not act like a full URL scanner or browser-security product.
